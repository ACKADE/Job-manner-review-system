# 命名规范与前后端协作标准

## 一、goctl 生成代码的命名规范

### 1.1 默认行为

go-zero 的 `goctl` 工具生成的文件名**默认使用全小写**（gozero）风格，这是 Go 语言官方推荐的命名方式，与 go-zero 框架的设计哲学一致。

### 1.2 可选的命名风格

goctl 通过 `--style` 参数控制生成的文件和文件夹命名风格。以下是可选的风格：

| 风格描述 | `--style` 参数值 | 命名效果示例 |
|---------|----------------|-------------|
| 全小写（默认） | `gozero` | `getjobhandler.go` |
| 驼峰式 | `goZero` | `getJobHandler.go` |
| 蛇形（下划线） | `go_zero` | `get_job_handler.go` |
| 大写 | `GOZERO` | `GETJOBHANDLER.GO` |

### 1.3 项目中的规范

**本项目统一使用 `gozero` 风格（全小写）**，原因如下：

1. **Go 官方推荐**：Go 语言规范建议文件和文件夹使用小写命名
2. **go-zero 框架风格**：保持与框架生成代码的一致性
3. **跨平台兼容性**：避免在大小写不敏感的文件系统中出现问题
4. **一致性**：统一命名风格降低认知负担

### 1.4 使用方式

在执行 goctl 命令时，**必须**添加 `--style gozero` 参数：

```bash
# 生成 API Handler（正确示例）
goctl api go -api api/career.api -dir . --style gozero

# 生成 RPC 代码（正确示例）
goctl rpc proto -src user.proto -dir . --style gozero
```

**禁止**使用其他风格，例如：

```bash
# ❌ 错误示例 - 不要使用驼峰命名
goctl api go -api api/career.api -dir . --style goZero

# ❌ 错误示例 - 不要使用蛇形命名
goctl api go -api api/career.api -dir . --style go_zero
```

### 1.5 生成的文件命名示例

使用 `--style gozero` 后，生成的文件将遵循以下模式：

| 功能模块 | Handler 文件 | Logic 文件 |
|---------|-------------|-----------|
| 获取岗位 | `getjobhandler.go` | `getjoblogic.go` |
| 创建学生 | `createstudenthandler.go` | `createstudentlogic.go` |
| 更新报告 | `updatereporthandler.go` | `updatereportlogic.go` |
| 删除岗位 | `deletejobhandler.go` | `deletejoblogic.go` |

## 二、代码命名规范

### 2.1 文件和文件夹命名

- **所有文件名**：全小写，单词间不加分隔符
- **所有文件夹名**：全小写，单词间不加分隔符

**正确示例**：
```
internal/handler/jobhandler.go
internal/logic/studentlogic.go
internal/model/users_model.go
```

**错误示例**：
```
❌ internal/handler/JobHandler.go
❌ internal/logic/StudentLogic.go
❌ internal/model/UsersModel.go
```

### 2.2 Go 代码中的命名

#### 结构体和接口

- **结构体**：使用 PascalCase（首字母大写）
- **接口**：使用 PascalCase（首字母大写）

```go
// 正确
type Student struct {
    ID       int64  `json:"id"`
    Name     string `json:"name"`
    Education string `json:"education"`
}

type UserService interface {
    GetByID(id int64) (*Student, error)
}

// 错误
type student struct { }
type user_service interface { }
```

#### 变量和函数

- **导出的变量/函数**：使用 PascalCase
- **未导出的变量/函数**：使用 camelCase

```go
// 正确
func GetAllStudents(ctx context.Context) ([]*Student, error) {
    var allStudents []*Student
    // ...
    return allStudents, nil
}

// 错误
func getAllStudents(ctx context.Context) ([]*Student, error) { } // 未导出
func get_all_students(ctx context.Context) ([]*Student, error) { } // 不符合 Go 规范
```

#### 常量

```go
// 正确
const (
    DefaultTimeout = 30
    MaxRetryCount  = 3
)

// 错误
const (
    DEFAULT_TIMEOUT = 30
    max_retry_count = 3
)
```

### 2.3 API 路由命名

#### RESTful 风格

```go
// 正确 - RESTful 风格，全小写
POST   /api/v1/students
GET    /api/v1/students/:id
PUT    /api/v1/students/:id
DELETE /api/v1/students/:id
GET    /api/v1/students/me
POST   /api/v1/match

// 错误
POST   /api/v1/Students
GET    /api/v1/GetStudents/:id
POST   /api/v1/match-student-job
```

#### 查询参数命名

查询参数使用 camelCase：

```
# 正确
GET /api/v1/students?page=1&pageSize=10&major=计算机科学
GET /api/v1/reports?studentId=1&status=completed

# 错误
GET /api/v1/students?Page=1&PageSize=10
GET /api/v1/reports?student_id=1
```

## 三、前后端协作标准

### 3.1 API 路由约定

#### 统一前缀和版本控制

- 所有 API 使用 `/api/v1` 前缀
- 使用 RESTful 风格的 URL

```
/api/v1/{resource}
/api/v1/{resource}/:id
/api/v1/{resource}/:id/action
```

#### 资源命名

资源使用**单数**形式：

```
# 正确
/api/v1/student
/api/v1/job
/api/v1/report

# 错误
/api/v1/students
/api/v1/jobs
/api/v1/reports
```

#### 特殊动作

对于非 CRUD 操作，使用 `/action` 格式：

```
POST /api/v1/students/resume              # 上传简历
POST /api/v1/reports/generate             # 生成报告
POST /api/v1/reports/polish               # 润色报告
POST /api/v1/match                        # 匹配人岗
GET  /api/v1/students/me                  # 获取当前用户
GET  /api/v1/reports/me                   # 获取我的报告
GET  /api/v1/jobs/:id/promotion-path      # 获取晋升路径
```

### 3.2 HTTP 方法约定

| 方法 | 用途 | 示例 |
|-----|------|------|
| GET | 查询资源 | `GET /api/v1/students/:id` |
| POST | 创建资源或执行操作 | `POST /api/v1/students` |
| PUT | 完整更新资源 | `PUT /api/v1/students/:id` |
| DELETE | 删除资源 | `DELETE /api/v1/students/:id` |

### 3.3 请求/响应格式

#### 统一响应格式

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    // 实际数据
  }
}
```

#### 状态码约定

| code | 含义 |
|-----|------|
| 0 | 成功 |
| 1001 | 参数错误 |
| 1002 | 认证失败 |
| 1003 | 权限不足 |
| 1004 | 资源不存在 |
| 500 | 服务器内部错误 |

#### JSON 字段命名

JSON 字段使用 **camelCase**：

```json
// 正确
{
  "studentId": 1,
  "userName": "zhangsan",
  "educationLevel": "本科",
  "graduationYear": 2025
}

// 错误
{
  "student_id": 1,
  "user_name": "zhangsan",
  "education_level": "本科"
}
```

### 3.4 认证机制

- 使用 JWT Token 进行认证
- Token 放在请求头 `Authorization: Bearer <token>`
- 以下接口无需认证：
  - `POST /api/v1/user/login`
  - `POST /api/v1/user/register`
  - `GET /health`
- 其他所有接口均需携带有效的 Token

### 3.5 错误处理

#### 错误响应格式

```json
{
  "code": 1002,
  "msg": "认证失败，请重新登录"
}
```

#### 常见错误场景

| 场景 | code | msg |
|-----|------|-----|
| Token 无效或过期 | 1002 | 认证失败，请重新登录 |
| 参数缺失或格式错误 | 1001 | 参数错误：缺少必填字段 'name' |
| 资源不存在 | 1004 | 资源不存在：学生 ID 123 不存在 |
| 权限不足 | 1003 | 权限不足：无权访问该资源 |
| 服务器内部错误 | 500 | 服务器内部错误，请稍后重试 |

### 3.6 流式接口规范

#### SSE (Server-Sent Events) 格式

流式接口使用 SSE 协议，数据格式如下：

```
data: {"type": "text", "content": "生成中..."}\n\n
data: {"type": "text", "content": "继续生成..."}\n\n
data: [DONE]\n\n
```

#### 前端处理

前端应使用 `EventSource` 或 `fetch` + `ReadableStream` 处理 SSE 响应：

```typescript
// 示例：使用 fetch 处理 SSE
const response = await fetch('/api/v1/reports/generate-stream', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({ studentId: 1, targetJobId: 1 })
});

const reader = response.body!.getReader();
const decoder = new TextDecoder();

while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  
  const chunk = decoder.decode(value);
  // 解析 SSE 数据
  const lines = chunk.split('\n');
  for (const line of lines) {
    if (line.startsWith('data:')) {
      const data = line.slice(5).trim();
      if (data === '[DONE]') break;
      const parsed = JSON.parse(data);
      // 处理数据
    }
  }
}
```

### 3.7 文件上传

#### 简历上传

简历文件需转换为 base64 字符串传递：

```json
{
  "fileContent": "base64编码的文件内容",
  "fileName": "resume.pdf"
}
```

#### 文件大小限制

- 简历文件：最大 10MB
- 支持格式：PDF、DOC、DOCX

### 3.8 分页参数

所有列表接口支持分页：

| 参数 | 类型 | 默认值 | 说明 |
|-----|------|-------|------|
| page | int | 1 | 页码（从 1 开始） |
| pageSize | int | 10 | 每页数量 |

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "list": [...],
    "total": 100,
    "page": 1,
    "pageSize": 10
  }
}
```

## 四、开发流程规范

### 4.1 API 定义优先

1. 在 `api/` 目录下定义或修改 `.api` 文件
2. 使用 `goctl api go` 生成代码（**必须使用 `--style gozero`**）
3. 在 `internal/logic/` 中实现业务逻辑
4. 在 `internal/handler/` 中实现 HTTP 处理（如需自定义）

### 4.2 代码生成命令

```bash
# 生成 API 代码
goctl api go -api api/career.api -dir . --style gozero

# 生成 Model 代码
goctl model mysql datasource -url="user:password@tcp(host:port)/dbname" -table="table_name" -dir ./internal/model --style gozero
```

### 4.3 代码审查检查点

在提交代码前，检查以下项目：

- [ ] 文件名是否全小写
- [ ] API 路由是否符合 RESTful 规范
- [ ] JSON 字段是否使用 camelCase
- [ ] 是否使用了 `--style gozero` 生成代码
- [ ] 错误码是否正确
- [ ] 是否添加了必要的认证检查
- [ ] 流式接口是否正确实现了 SSE 格式

## 五、常见问题

### Q1: 为什么强制使用全小写文件名？

**A**: 
1. Go 语言官方推荐
2. go-zero 框架默认风格
3. 避免跨平台兼容性问题
4. 保持代码库的一致性

### Q2: 为什么 JSON 字段使用 camelCase 而不是 snake_case？

**A**:
1. JavaScript/TypeScript 前端使用 camelCase
2. 前后端命名风格一致，减少转换成本
3. 符合现代 Web API 的最佳实践

### Q3: 可以混合使用不同的命名风格吗？

**A**: **不可以**。项目必须保持统一的命名风格，这有助于代码的可维护性和可读性。

### Q4: 如果 goctl 生成代码后又手动修改了文件名怎么办？

**A**: **不要手动修改 goctl 生成的文件名**。如果需要自定义逻辑，应该在 Logic 层实现，而不是修改 Handler 文件名。如果必须修改，请确保所有引用都同步更新。

## 六、工具配置

### 6.1 Git 钩子（可选）

建议在项目中添加 pre-commit 钩子，自动检查 Go 文件的命名规范：

```bash
#!/bin/bash
# .git/hooks/pre-commit
# 只检查 Go 文件的大写命名规范

# 获取暂存区中的所有 .go 文件
GO_FILES=$(git diff --cached --name-only | grep '\.go$')

# 如果有 .go 文件，检查是否包含大写字母
if [ -n "$GO_FILES" ]; then
    if echo "$GO_FILES" | grep -E '[A-Z]'; then
        echo "错误：检测到大写文件名，Go 文件必须使用全小写命名"
        echo "请修改以下文件名："
        echo "$GO_FILES" | grep -E '[A-Z]' | sed 's/^/  - /'
        exit 1
    fi
fi
```

**说明**：该钩子只检查 `.go` 文件的命名规范，不会检查其他类型文件（如 `.tsx`, `.md` 等）。

### 6.2 IDE 配置

建议配置 IDE 自动格式化工具：

- Go: 使用 `gofmt` 自动格式化
- TypeScript/JavaScript: 使用 ESLint + Prettier
